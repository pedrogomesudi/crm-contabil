# V6.3 — Contas a Pagar + Estorno Auditado — Design

**Data:** 2026-07-04
**Marco:** V6 — Módulo Financeiro · **Sub-marco:** V6.3 (fecha o módulo)
**Depende de:** V6.1 (cadastros: fornecedor/categoria/centro_custo/conta_bancaria), V6.2 (titulo/baixa/status), V6.5 (dashboard).
**Requisitos-fonte:** `módulo financeiro.docx` — RF-023, RF-024, RF-025, RF-027, RF-028 (parcial), RNF-001/002.

## 1. Objetivo

Fechar o módulo financeiro com o lado **despesa**: lançar contas a pagar (única, parcelada, recorrente) com anexos, dar **baixa (pagamento)**, **estornar** baixas com trilha auditada, e estender o dashboard para o **saldo real** (entradas − saídas). Aprovação de pagamento (RF-026) fica para um incremento futuro.

## 2. Fronteiras (escopo)

**Dentro:**
- Extensão do `titulo` para RECEBER/PAGAR (+ fornecedor, parcelamento).
- **Contas a pagar:** despesa única, parcelada (N parcelas), recorrente (template + geração mensal).
- **Baixa (pagamento)** reusando `baixa`; **estorno auditado** (justificativa; substitui o "desfazer baixa" da V6.2, em receber e pagar).
- **Anexos** por título (NF/boleto/comprovante) via storage.
- **Dashboard** com o lado despesa: saídas, a pagar, **saldo real**, aging de pagar, receita × despesa.
- Tela `/financeiro/contas-a-pagar`.

**Fora (incrementos futuros):**
- **Aprovação de pagamento** acima de limite (RF-026).
- Receita eventual/servico (RF-016), reajuste (RF-018), conciliação bancária (RF-035), DRE completa (RF-040).

## 3. Modelo de dados

Convenções do módulo (numeric(15,2); competência 1º dia; auditoria; sem exclusão física).

### 3.1 Enums
- Novo `titulo_tipo` : `RECEBER | PAGAR`.
- `titulo_origem` **ganha** `DESPESA_AVULSA`, `DESPESA_PARCELADA`, `DESPESA_RECORRENTE`.

### 3.2 Extensão de `titulo`
`alter table titulo`:
- `tipo titulo_tipo not null default 'RECEBER'` (títulos atuais → RECEBER).
- `cliente_id` passa a **nulável** (`drop not null`).
- `fornecedor_id uuid references fornecedor(id)`.
- `parcela smallint`, `total_parcelas smallint`, `grupo_parcelamento_id uuid`.
- Check: `(tipo='RECEBER' and cliente_id is not null) or (tipo='PAGAR' and fornecedor_id is not null)`.
- Idempotência das despesas recorrentes: `unique (grupo_parcelamento_id, competencia) where origem='DESPESA_RECORRENTE'` (o grupo aponta o template). Únicas/parceladas não têm restrição de idempotência (cada lançamento é explícito).

### 3.4 `baixa` — estorno auditado
`alter table baixa add`: `estornada boolean not null default false`, `estorno_motivo text`, `estorno_em timestamptz`, `estorno_por uuid references usuarios(id)`.
O trigger `recalcular_status_titulo` passa a **somar só baixas com `estornada = false`**. Estornar = `update baixa set estornada=true, estorno_motivo=…, estorno_em=now(), estorno_por=…` → o trigger recomputa o status (nunca deleta a baixa — trilha preservada, RNF-002).

Bucket de storage próprio (`anexos-financeiro`) ou reuso do bucket de documentos com prefixo; upload via **service_role** (server action), como os documentos.

## 4. Contas a pagar (fluxo)

Tela `/financeiro/contas-a-pagar` (gate `podeGerenciarFinanceiro`).

- **Lançar despesa:** fornecedor, categoria (despesa), centro de custo (opcional), descrição, valor, 1º vencimento, competência. Modo:
  - **Única** → 1 título `PAGAR` origem `DESPESA_AVULSA`.
  - **Parcelada (N)** → N títulos `DESPESA_PARCELADA`, `grupo_parcelamento_id` comum, `parcela`/`total_parcelas` (1..N), valor = round(total/N, 2) com ajuste do centavo na última, vencimento = 1º + (i−1) meses, competência idem.
  - **Recorrente** → cria `despesa_recorrente` (template); as parcelas mensais nascem na geração.
- **Gerar recorrentes:** RPC `gerar_despesas_recorrentes(p_competencia date)` — para cada `despesa_recorrente` ativa com `data_inicio ≤ fim da competência`, cria um título `PAGAR`/`DESPESA_RECORRENTE` (idempotente pelo unique §3.2). Botão "Gerar despesas do mês" na tela. **A automação mensal** (`gerar_mensalidades_automatico`, pg_cron) passa a chamar **também** essa RPC quando o flag está on.
- **Baixa (pagamento):** reusa `baixa` (conta de saída, forma, juros/multa/desconto). Status via trigger.
- **Estorno:** `estornarBaixa(baixaId, motivo)` (justificativa obrigatória) — marca estornada; recomputa status. Vale para receber e pagar.

## 5. Anexos
- Na ficha do título (tela contas a pagar): upload de arquivo → server action grava no storage (service_role) + linha em `anexo_titulo`; listar, baixar (URL assinada) e remover. Gate `podeGerenciarFinanceiro`.

## 6. Dashboard (extensão)
`financeiro_dashboard(competencia)` passa a retornar também:
- `saidas_mes` = Σ baixas **não-estornadas** de títulos PAGAR com `data_recebimento` na competência.
- `a_pagar_mes` = Σ saldo dos títulos PAGAR da competência.
- `saldo_real` = Σ saldo_inicial contas + Σ recebimentos (RECEBER, não-estornados) − Σ pagamentos (PAGAR, não-estornados).
- `receita_despesa` = { receita, despesa } do mês (títulos por tipo).
`financeiro_aging(p_tipo titulo_tipo)` — passa a aceitar o tipo (aging de receber e de pagar). UI mostra os dois. O **aviso "só entradas" sai**.

## 7. Permissões e RLS
- **`titulo` (V6.2) NÃO muda de policy:** o SELECT do contador já exige `exists (clientes where id = titulo.cliente_id and contador_id = auth.uid())` — como PAGAR tem `cliente_id` nulo, o contador **não casa e não vê** contas a pagar automaticamente. Escrita já é admin/financeiro. Nada a ajustar.
- **Novas tabelas** `despesa_recorrente` e `anexo_titulo`: RLS `auth_papel() in ('admin','financeiro')` (SELECT/INSERT/UPDATE). Contador e assistente não veem.
- Estorno de baixa: `podeGerenciarFinanceiro` (a baixa já é admin/financeiro na V6.2).

## 8. UI
- Nova tela `/financeiro/contas-a-pagar`: lançar despesa (form com modo único/parcelado/recorrente), lista de títulos PAGAR por competência, baixar, estornar, anexos, gerenciar despesas recorrentes.
- Tela `/financeiro/contas-a-receber`: trocar "Desfazer baixa" por **"Estornar"** (com motivo).
- Dashboard: novos cards (saldo real, saídas, a pagar, receita × despesa) + aging de pagar; remover o aviso "só entradas".
- Link "Contas a pagar" no hub `/financeiro/cadastros`.

## 11. Riscos e observações
- **`cliente_id` nulável em `titulo`:** revisar consumidores (V6.2/V6.5) que assumem cliente — a geração de mensalidades e as RPCs de relatório filtram por tipo/origem; a tela de receber lista RECEBER. Adicionar `tipo='RECEBER'`/`'PAGAR'` nos filtros das telas e RPCs para não misturar.
- **Estorno substitui "desfazer baixa":** a action `desfazerBaixa` (V6.2) é trocada por `estornarBaixa`; garantir que o trigger de status ignore estornadas (senão o saldo fica errado).
- **Automação:** o job pg_cron passa a gerar receber + pagar; idempotência protege reexecução.
- **Saldo real** continua sem conciliação bancária (V6.2+); é saldo de caixa por lançamentos, não extrato conciliado.
