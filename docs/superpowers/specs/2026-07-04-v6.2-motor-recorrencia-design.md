# V6.2 — Motor de Recorrência (Contas a Receber) — Design

**Data:** 2026-07-04
**Marco:** V6 — Módulo Financeiro · **Sub-marco:** V6.2
**Depende de:** V6.1 (cadastros: `conta_bancaria`, `categoria`, `centro_custo`; extensão `clientes_financeiro`).
**Requisitos-fonte:** `módulo financeiro.docx` — RF-010 a RF-017, RF-020 a RF-023 (parcial), RF-013 (pró-rata), RNF-001/002/005/010.

## 1. Objetivo

Entregar o ciclo de **contas a receber recorrente ponta a ponta**: o **contrato de honorários** (na ficha do cliente) → **geração automática das mensalidades** (+ 13º + pró-rata) como títulos de contas a receber → **baixa (recebimento)** lançada no financeiro (com a conta bancária do V6.1). É o coração do módulo financeiro.

## 2. Fronteiras (escopo)

**Dentro do V6.2:**
- Entidade **contrato** (por cliente, na ficha; pode haver mais de um).
- Entidade **titulo** (conta a receber) + **baixa** (recebimento).
- **Motor de geração** das mensalidades: por contrato, com **pró-rata** (RF-013) e **13º**, **idempotente**.
- **Disparo manual** ("Gerar mensalidades do mês") + **automação** (pg_cron mensal, ligável por toggle).
- **Baixa (recebimento)** total/parcial gravando entrada na conta bancária; **desfazer baixa** (correção simples).
- **Tela Contas a Receber** + seção de contrato na ficha do cliente.
- **Sincronização** contrato → `clientes_financeiro.honorario_mensal` (para a NFS-e seguir o contrato sem mudar o caminho fiscal).

**Fora do V6.2 (marcos seguintes):**
- **Estorno auditado completo** de baixa, **contas a pagar**, aprovação → V6.3.
- **Reajuste em lote** (RF-018/019) → marco próprio.
- **Régua de cobrança** → V6.4. **Relatórios/dashboard** (fluxo de caixa, aging, MRR) → V6.5.

**Restrição inegociável:** a **emissão de NFS-e avulsa** (valor/descrição próprios, fora do honorário mensal — já em produção) **não pode ser alterada**. A sincronização só escreve `honorario_mensal` (valor recorrente); a avulsa usa valor digitado à parte e permanece intacta.

## 3. Modelo de dados

Convenções (iguais ao V6.1): PK `uuid default gen_random_uuid()`; monetário `numeric(15,2)`; `criado_em/por`, `atualizado_em/por`; competência = 1º dia do mês; sem exclusão física (status/cancelamento).

### 3.1 Novos enums
```
contrato_status      : ATIVO | SUSPENSO | ENCERRADO
indice_reajuste      : IPCA | IGPM | INPC | PERCENTUAL_FIXO | SEM_REAJUSTE
titulo_origem        : MENSALIDADE | DECIMO_TERCEIRO
titulo_status        : ABERTO | VENCIDO | BAIXADO | BAIXADO_PARCIAL | CANCELADO
forma_pagamento      : PIX | BOLETO | CARTAO | TRANSFERENCIA | DINHEIRO
```

|---|---|---|

**Idempotência (RNF-010):** índice único parcial
`unique (contrato_id, competencia, origem) where contrato_id is not null`
e `unique (cliente_id, competencia, origem) where contrato_id is null` (título do honorário).

### 3.4 `baixa` (recebimento)
| Campo | Tipo | Regra |
|---|---|---|
| id | uuid PK | |
| titulo_id | uuid → titulo not null | |
| data_recebimento | date not null | |
| valor_recebido | numeric(15,2) not null | |
| juros / multa / desconto | numeric(15,2) default 0 | |
| conta_bancaria_id | uuid → conta_bancaria not null | conta de entrada (V6.1) |
| forma_pagamento | forma_pagamento not null | |
| auditoria | (padrão) | |

Regra: soma das baixas de um título ≤ `valor + juros + multa`. Um título pode ter várias baixas (parciais). **Desfazer baixa** = deletar a baixa e recomputar o status do título (V6.2 usa correção simples; estorno auditado completo é V6.3).

### 3.5 Sincronização contrato → honorário
Trigger em `contrato` (após insert/update/delete): recomputa
`clientes_financeiro.honorario_mensal = coalesce(sum(valor_mensal) filter (status='ATIVO'), 0) → nullif(…, 0)`
para o `cliente_id` afetado. Assim a **NFS-e (que lê `honorario_mensal`) segue o contrato** automaticamente, sem tocar no caminho fiscal. Cliente sem contrato mantém o honorário atual (o trigger só dispara em mudança de contrato). Encerrar todos os contratos ⇒ honorário zera.

## 4. Motor de geração

**Universo:** clientes **ativos, não-excluídos** com **contrato ATIVO** OU `honorario_mensal > 0`.

**Regra por cliente/contrato, para a competência C (1º dia do mês):**
- Para cada **contrato ATIVO** cujo `data_inicio ≤ fim de C`: gera título `MENSALIDADE` com `valor = valor_mensal`, `vencimento = C + dia_vencimento`, `categoria_id/centro_custo_id` do contrato.
- Cliente **sem contrato** mas com `honorario_mensal > 0`: gera **um** título `MENSALIDADE` a partir do honorário + `clientes_financeiro.dia_vencimento` (fallback: dia 10). `categoria_id` = "Honorários mensais" (seed do V6.1).
- **Pró-rata (RF-013):** se `data_inicio` cai **dentro** do mês de C (primeiro mês do contrato), `valor = round(valor_mensal × dias_restantes / dias_do_mês, 2)`, onde `dias_restantes = (último dia de C − data_inicio + 1)`. Nos meses seguintes, valor cheio.
- **13º:** para contratos com `gera_decimo_terceiro`, quando `mês(C) = mes_decimo_terceiro`, gera título `DECIMO_TERCEIRO` (`valor = valor_mensal`, vencimento = C + dia_vencimento).

**Idempotência:** a inserção usa `on conflict do nothing` sobre os índices únicos de §3.3 — reexecutar a geração do mesmo mês não duplica.

**Encerramento/suspensão de contrato:** ao encerrar (ou suspender) um contrato, **cancela** (`status = CANCELADO`) os títulos **futuros em ABERTO** daquele contrato (competência ≥ mês corrente); títulos vencidos/baixados permanecem.

**Disparo:**
- **Manual:** botão "Gerar mensalidades do mês" (escolhe a competência) → server action (gate admin/financeiro) → RPC `gerar_mensalidades(competencia)`. Retorna quantos títulos criados/pulados.
- **Automático:** flag `financeiro_config.geracao_automatica` (boolean) + job **pg_cron** mensal que chama a RPC para a competência corrente **quando o flag está ligado**. Toggle na tela liga/desliga o flag.

## 5. Baixa (recebimento) e correção
- **Registrar baixa:** por título — `data_recebimento`, `valor_recebido`, `conta_bancaria_id`, `forma_pagamento`, opcional juros/multa/desconto. Recalcula o status do título: soma das baixas ≥ (valor+juros+multa) ⇒ `BAIXADO`; parcial > 0 ⇒ `BAIXADO_PARCIAL`. Valida que não excede o total.
- **Desfazer baixa:** remove a baixa e recomputa o status (volta a ABERTO/BAIXADO_PARCIAL). Correção simples; estorno auditado completo é V6.3.

## 6. UI

**Ficha do cliente** (`clientes/[id]`): nova seção **"Contratos"** (gate `podeVerHonorario`): lista os contratos do cliente + form de criar/editar/encerrar. Ao salvar, o trigger sincroniza o honorário (a seção de honorário passa a refletir a soma dos contratos).

**Tela "Contas a receber"** (`/financeiro/contas-a-receber`, gate `podeVerHonorario`):
- Filtros: competência (mês), status, cliente.
- Botão **"Gerar mensalidades do mês"** + **toggle de automação** (gate `podeGerenciarFinanceiro`).
- Lista de títulos (cliente, origem, competência, vencimento, valor, status derivado, saldo).
- Ação **Baixar** por título (form: data, valor, conta, forma, juros/multa/desconto) e **Desfazer baixa**.

Entrada no menu "Financeiro" (já existe do V6.1) ganha "Contas a receber".

## 7. Permissões e RLS
- **Contrato:** SELECT/INSERT/UPDATE `admin|financeiro` e `contador` do próprio cliente (`clientes.contador_id = auth.uid()`) — mesma regra de `clientes_financeiro`. `podeVerHonorario` no gate de servidor.
- **titulo / baixa:** SELECT `admin|financeiro|contador-dono`; INSERT/UPDATE (geração, baixa) `admin|financeiro` (`podeGerenciarFinanceiro`). Assistente: sem policy (não vê nada financeiro).
- **RPC `gerar_mensalidades`:** `security definer` (para o pg_cron rodar sem sessão) com checagem interna de competência; a versão chamada pelo botão valida papel no server action antes.
- Sem exclusão física (RNF-002): contrato encerra, título cancela; baixa pode ser desfeita (V6.2) / estornada (V6.3).

## 8. Testes
- **Unitários (puros):** cálculo de **pró-rata** (dias no mês, data_inicio no meio/começo/fim), decisão de **13º** (mês bate), `vencimento` a partir de competência+dia, recomputo de **status do título** a partir das baixas.
- **RLS (`rls.test.sql`):** matriz admin/financeiro/contador/assistente para `contrato/titulo/baixa` (contador vê só os seus; assistente nada).
- **RPC de geração (`db:test`):** idempotência (reexecução não duplica), pró-rata no 1º mês, 13º no mês certo, encerramento cancela títulos futuros ABERTO, universo (contrato vs honorário).
- **Trigger de sync:** salvar/encerrar contrato ajusta `honorario_mensal` (= soma ativos).
- Antes de commitar: `npm run lint && npm run typecheck && npm test && npm run db:migrate && npm run db:test`.

## 10. Riscos e observações
- **pg_cron:** exige habilitar a extensão no Supabase e agendar o job; a automação depende disso. O botão manual funciona independente (fallback). Se `pg_cron` não estiver disponível, a automação fica como toggle inerte até habilitar (documentar).
- **Sync do honorário × NFS-e:** o trigger muda `honorario_mensal`; validar que a NFS-e recorrente e a avulsa continuam corretas (a avulsa não lê esse campo para o valor).
- **Múltiplos contratos × 1 NFS-e:** o financeiro gera um título por contrato; a NFS-e recorrente emite uma nota pelo total (honorário = soma). São visões diferentes (recebível por contrato × documento fiscal agregado) — comportamento aceito e documentado.
- **Múltiplas baixas parciais:** somatório validado no servidor; status derivado do total baixado.
