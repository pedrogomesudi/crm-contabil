# V7.3 — Atendimento / Inbox WhatsApp — Design

**Data:** 2026-07-04
**Marco:** V7 — Integração WhatsApp · **Sub-marco:** V7.3 (atendimento — fecha a V7)
**Depende de:** V7.1 (cliente Z-API, config cifrada, `whatsapp_mensagem`), V7.2 (régua — saídas na mesma tabela).
**Requisitos-fonte:** roadmap V7.3 (atendimento bidirecional).

## 1. Objetivo

Caixa de entrada bidirecional de WhatsApp: **receber** mensagens dos clientes (webhook do Z-API) e **responder** por uma inbox de dois painéis, com a conversa **unificada** (recebidas + respostas + cobranças/régua) por contato.

## 2. Fronteiras (escopo)

**Dentro:** webhook de entrada (dedup + resolução best-effort do cliente), extensão de `whatsapp_mensagem` para direção/lida/z_message_id, conversa derivada por telefone, tela `/atendimento` (lista + thread + resposta), marcação de lidas, atualização por polling.

**Fora (futuro):** Realtime, mídia/áudio/imagem (só texto aqui), respostas automáticas/bot, atribuição de atendente/filas, notificações push, busca na conversa, envio a partir de um contato novo digitado (só responde a quem existe na thread).

**Ressalvas:** herda o Z-API não-oficial. Só **texto** (mensagens de mídia recebidas são registradas como "[mídia não suportada]" para não perder o contexto, mas sem baixar o conteúdo).

## 3. Modelo de dados

### 3.1 Extensão de `whatsapp_mensagem`
- Enum `whatsapp_status` **ganha** `RECEBIDO`.
- `direcao` enum `whatsapp_direcao` (`IN`|`OUT`) not null default `'OUT'` (as existentes = saídas).
- `lida` boolean not null default true (entradas novas nascem `false`).
- `z_message_id` text — id da mensagem no Z-API; **unique** (parcial `where z_message_id is not null`) para deduplicar o webhook.
- Índice `(telefone, criado_em)` para montar a thread.
A **conversa** é derivada: agrupamento por `telefone`. Não há tabela `conversa` — a thread é o conjunto de `whatsapp_mensagem` daquele telefone.

## 4. Webhook de entrada

- Rota `POST /api/webhooks/zapi/[secret]` (segredo no path, comparado a `ZAPI_WEBHOOK_SECRET` em tempo constante — padrão do webhook Clicksign). Service_role.
- **Parser puro** `extrairMensagemZapi(payload): { telefone: string; texto: string; zId: string } | null` — extrai do payload do Z-API (`phone`, `text.message`/`message`, `messageId`). Regra: **exige `phone` e `messageId`**; `texto` = a mensagem de texto, ou `"[mídia não suportada]"` quando é uma mensagem recebida sem texto (mídia); retorna **null** apenas para eventos que **não são mensagem recebida** (status de entrega, ack, mensagens enviadas por nós — `fromMe=true`).
- Fluxo: valida segredo → `extrairMensagemZapi` → se null, `200 {ignored}` → normaliza telefone → **dedup** por `z_message_id` (se já existe, `200 {duplicado}`) → resolve `cliente_id` por telefone (best-effort; múltiplos/none → null) → insere `direcao=IN, status=RECEBIDO, lida=false, z_message_id`. Sempre responde `200` rápido (webhooks não devem falhar por erro de negócio).

## 5. Inbox (leitura/escrita)

### 5.1 Helpers puros (`src/lib/whatsapp/inbox.ts`)
- `type MsgConversa = { telefone: string; texto: string; direcao: "IN" | "OUT"; lida: boolean; criado_em: string; cliente?: string | null }`.
- `agruparConversas(msgs: MsgConversa[]): Conversa[]` — agrupa por telefone; cada `Conversa = { telefone; cliente: string | null; ultima: string; ultima_em: string; nao_lidas: number }`; ordena por `ultima_em` desc.

### 5.2 Server actions (`/atendimento/actions.ts`)
- `listarConversas(): Conversa[]` — lê `whatsapp_mensagem` (via server client, RLS aplica), agrupa com `agruparConversas`.
- `abrirConversa(telefone): MsgConversa[]` — mensagens do telefone em ordem; **marca as entradas como lidas** (`update ... set lida=true where telefone=? and direcao='IN' and lida=false`).
- `responder(telefone, texto)` — decifra config (admin), `enviarTexto` (Z-API), insere `direcao=OUT, status`. Gate `podeAtender`.

## 6. UI
- **`/atendimento`** (gate `podeAtender` = admin/financeiro/contador): dois painéis. Esquerda: lista de conversas (contato/cliente, prévia da última, badge de não-lidas). Direita: thread com balões **IN** (esquerda) × **OUT** (direita) e caixa de resposta. **Polling** (~15s) refaz a lista e a thread aberta; botão "atualizar".
- Link "Atendimento" no menu principal (não no hub financeiro — atendimento é transversal).

## 7. Permissões e RLS
- Novo helper `podeAtender(papel) = ['admin','financeiro','contador'].includes(papel)`. Assistente fora.
- RLS de `whatsapp_mensagem` **não muda**: admin/financeiro tudo; contador só `cliente_id` dos seus; entradas de números sem cliente casado ficam visíveis a admin/financeiro (contador não vê número solto). O webhook (service_role) insere sem depender de sessão.
- Resposta: `podeAtender`; usa a config Z-API cifrada (admin/service_role).

## 8. Testes
- **Unit (`src/tests/whatsapp/inbox.test.ts`):** `extrairMensagemZapi` (payload com texto → objeto; evento de status/sem texto → null; extrai `zId`); `agruparConversas` (agrupa por telefone, conta não-lidas, ordena por recência).
- **RLS (`db:test`):** contador vê thread só do seu cliente; assistente nada; dedup por `z_message_id` (2ª inserção com mesmo id barra pelo unique).
- **Manual:** configurar o webhook no Z-API; enviar do celular para o número → aparece na inbox; responder → chega no celular; conferir marcação de lida e a thread unificada (aparece a cobrança/régua junto).
- Antes de commitar: `npm run lint && npm run typecheck && npm test && npm run db:migrate && npm run db:test`.

## 10. Riscos e observações
- **Webhook confiável:** o Z-API precisa estar configurado para postar no endpoint; sem isso, não há entrada (a saída/cobrança segue funcionando). Documentar no README.
- **Dedup:** `z_message_id` único evita duplicar reentregas do webhook.
- **Número → cliente:** resolução best-effort por telefone normalizado; número não cadastrado aparece como contato "sem cliente" (admin/financeiro). Se dois clientes tiverem o mesmo telefone, resolve como não-casado (null) para não atribuir errado.
- **Polling:** custo baixo (dezenas de conversas); Realtime fica como melhoria futura.
- **Só texto:** mídias recebidas viram um marcador textual; envio de mídia fora de escopo.
- **LGPD:** as conversas são dados pessoais; acesso restrito por RLS/papel; retenção segue a política geral do sistema.
