# V6.5 — Relatórios e Dashboard (contas a receber) — Design

**Data:** 2026-07-04
**Marco:** V6 — Módulo Financeiro · **Sub-marco:** V6.5
**Depende de:** V6.1 (cadastros), V6.2 (contrato/titulo/baixa).
**Requisitos-fonte:** `módulo financeiro.docx` — RF-039, RF-041, RF-042, RF-043, RF-044, RF-048 a RF-052 (parcial, lado receita).

## 1. Objetivo

Entregar um **dashboard financeiro** e os relatórios gerenciais do lado **receita** (contas a receber), a partir dos dados que já existem (contratos, títulos, baixas, clientes, contas bancárias): saldo, MRR, recebido/a receber, inadimplência, aging, fluxo de caixa e receita por tipo.

## 2. Fronteiras (escopo)

**Dentro:**
- Página **`/financeiro/dashboard`** (cards + tabelas + barras CSS; sem biblioteca de gráficos).
- RPCs de agregação: `financeiro_dashboard`, `financeiro_aging`, `financeiro_fluxo_caixa`.
- Indicadores: saldo, MRR, recebido no mês, a receber, inadimplência (R$ e %), previsão 30/60/90.
- Relatórios: aging de contas a receber, maiores devedores, fluxo de caixa (6 meses), receita por tipo.

**Fora (marcos seguintes):**
- **Contas a pagar / despesas / DRE / orçado×realizado** → dependem da V6.3; ficam para depois.
- Evolução histórica de MRR com snapshots (só temos estado atual) → não incluído.
- Exportação PDF/Excel dos relatórios (RF-047) → depois.
- Gráficos com biblioteca → decisão de manter barras CSS.

**Ressalva documentada:** sem contas a pagar, o "saldo" e o "fluxo de caixa" refletem **apenas entradas** (saldo inicial das contas + recebimentos). Deixar explícito na tela.

## 3. Arquitetura e cálculo

Segue o padrão do dashboard existente (home `/` usa a RPC `dashboard_resumo()` + `CardResumo`). RPCs **SECURITY INVOKER** para a **RLS escopar o contador** aos seus clientes (admin/financeiro veem tudo; assistente não vê nada — sem policy). Todas retornam `jsonb`.

Convenções: competência = 1º dia do mês; monetário `numeric(15,2)`; "hoje" = `current_date`.

### 3.1 `financeiro_dashboard(p_competencia date) returns jsonb`
Retorna:
- `saldo` = Σ `conta_bancaria.saldo_inicial` (ativas) + Σ `baixa.valor_recebido`.
- `mrr` = Σ `contrato.valor_mensal` (status ATIVO) + Σ `clientes_financeiro.honorario_mensal` dos clientes **ativos sem contrato ATIVO**.
- `recebido_mes` = Σ `baixa.valor_recebido` com `data_recebimento` na competência.
- `a_receber_mes` = Σ (`titulo.valor` − baixado) dos títulos da competência com status ≠ CANCELADO.
- `inadimplencia_total` = Σ saldo dos títulos **vencidos** (`vencimento < current_date`, saldo > 0, status ≠ CANCELADO/BAIXADO).
- `inadimplencia_pct` = `inadimplencia_total` / (carteira em aberto total) × 100 (0 se carteira 0).
- `previsao_30/60/90` = Σ saldo dos títulos em aberto com `vencimento` dentro de 30/60/90 dias a partir de hoje.

### 3.2 `financeiro_aging() returns jsonb`
Faixas por saldo em aberto (títulos não cancelados/baixados), classificadas por `vencimento` × `current_date`:
`a_vencer` (venc ≥ hoje) · `d1_30` · `d31_60` · `d61_90` · `d90_mais`. Cada uma com `{ total, qtd }`.

### 3.3 `financeiro_fluxo_caixa(p_meses int default 6) returns jsonb`
Array dos últimos `p_meses` meses (do mês corrente para trás, incluindo o corrente): `{ mes, realizado, a_receber }` — `realizado` = Σ baixas com `data_recebimento` no mês; `a_receber` = Σ saldo dos títulos em aberto com `vencimento` no mês.

### 3.4 Maiores devedores e receita por tipo
- **Maiores devedores:** query direta (respeita RLS) — clientes com títulos vencidos, `Σ saldo vencido` e `qtd`, ordenado desc, top 10.
- **Receita por tipo** (competência): Σ `titulo.valor` por `origem` (MENSALIDADE, DECIMO_TERCEIRO).

## 4. UI (`/financeiro/dashboard`)

- Gate `podeVerHonorario`; redirect se não.
- **Filtro** de competência (mês) no topo (default: mês corrente).
- **Cards** (reusa `CardResumo`): Saldo · MRR · Recebido no mês · A receber · Inadimplência (R$ + %) · Previsão 30/60/90.
- **Aging:** barras horizontais CSS por faixa + tabela (faixa, total, qtd).
- **Maiores devedores:** tabela (cliente, total vencido, nº títulos).
- **Fluxo de caixa (6 meses):** barras por mês (realizado × a receber) + valores.
- **Receita por tipo:** tabela/barra (mensalidade × 13º).
- **Aviso** de escopo: "valores refletem apenas entradas (contas a pagar entram numa próxima fase)".
- Acesso pelo hub `/financeiro/cadastros` e/ou Sidebar → "Dashboard financeiro".

## 5. Componentização
- SQL: 3 RPCs focadas (§3.1–3.3) + 2 queries (maiores devedores, receita por tipo) direto na page/action.
- TS puro: `src/lib/financeiro/relatorios.ts` — `faixaAging(diasAtraso): string`, `LABEL_FAIXA`, `pctInadimplencia(vencido, carteira): number`, formatação de mês. Testável.
- UI: `src/components/financeiro/DashboardFinanceiro.tsx` (client, filtro + render) + `page.tsx` (RSC busca inicial).

## 6. Permissões e RLS
- Página: `podeVerHonorario` (admin/financeiro/contador-dono).
- RPCs SECURITY INVOKER → RLS de `titulo/baixa/contrato/clientes_financeiro/conta_bancaria` aplica o escopo. Contador vê só os seus; assistente nada.

- **RPCs (`db:test`):** com contratos/títulos/baixas semeados — MRR (contrato + honorário), recebido no mês, inadimplência (vencido), aging por faixa, previsão. Cenários com datas fixas.

## 9. Riscos e observações
- **Saldo/fluxo só de entradas** enquanto não houver contas a pagar (V6.3) — comunicado na UI para não induzir a erro.
- **MRR é estado atual** (soma dos contratos/honorários ativos), não série histórica — sem snapshots, não há evolução mês a mês nesta fatia.
- **Performance:** RPCs agregadas sobre a base atual (dezenas a centenas de títulos) — trivial; índices de `titulo(competencia)` e `titulo(vencimento)` ajudam (o de competência já existe; adicionar `idx_titulo_vencimento`).
