# V6.1 — Fundação Financeira (Cadastros básicos) — Design

**Data:** 2026-07-03
**Marco:** V6 — Módulo Financeiro · **Sub-marco:** V6.1 (primeira fatia)
**Requisitos-fonte:** `módulo financeiro.docx` — RF-001 a RF-009, RF-006/RF-007 (extensão do cliente), RNF-001/002/003/005.

## 1. Contexto e decomposição da V6

A V6 (Módulo Financeiro) é grande demais para um único ciclo spec → plan → implement. Ela foi
decomposta em cinco sub-marcos, cada um com seu próprio ciclo (como a V5 foi fatiada em A/B/lote/
avulsa/cancelamento):

| Sub-marco | Escopo | Depende de |
|---|---|---|
| **V6.1 — Fundação financeira** | Cadastros de apoio + extensão financeira do cliente + permissões + RLS + telas | — |
| V6.2 — Motor de recorrência | Contratos, geração idempotente de mensalidades + 13º, títulos a receber (**pró-rata / RF-013 entra aqui**) | V6.1 |
| V6.3 — Baixas, estornos e contas a pagar | Baixa total/parcial, estorno com auditoria, títulos a pagar, aprovação | V6.2 |
| V6.4 — Régua de cobrança | Etapas parametrizáveis, e-mail/WhatsApp (casa com V7) | V6.3 |
| V6.5 — Relatórios e dashboard | Fluxo de caixa, aging, inadimplência, MRR, churn | V6.2+ |

Este documento cobre **apenas a V6.1**.

## 2. Objetivo do V6.1

Entregar a base de dados e as telas dos **cadastros de apoio** do financeiro, sobre a qual o motor de
recorrência (V6.2) e os títulos (V6.3) serão construídos. É uma fatia vertical completa (schema + RLS
+ permissões + Server Actions + UI + testes), coerente e testável isoladamente, sem depender de
nenhum dado transacional.

## 3. Fronteiras (escopo)

**Dentro do V6.1:**
- 5 cadastros de apoio: contas bancárias, plano de contas (categorias), centros de custo,
  fornecedores, tabela de serviços.
- Extensão financeira do cliente — **apenas os campos que ainda não existem**.
- Módulo de permissão financeira (`src/lib/financeiro/permissoes.ts`).
- RLS por tabela + telas CRUD + testes (RLS e unitários).

**Fora do V6.1 (sub-marcos seguintes):**
- Contratos e títulos (V6.2); baixas/estornos/contas a pagar (V6.3); régua (V6.4);
  relatórios/dashboard (V6.5).
- **Pró-rata (RF-013) → V6.2.**
- **`log_auditoria` completo (RNF-001)** e **fechamento de período (RNF-004)** → entram junto com os
  dados transacionais (títulos/baixas), onde a auditoria de valores realmente importa. No V6.1 vale o
  padrão já vigente no projeto: colunas `criado_por/criado_em`, `atualizado_por/atualizado_em` e
  **inativação lógica** (`ativa`), sem exclusão física (atende RNF-002 para os cadastros).
- Status financeiro do cliente `SUSPENSO/ENCERRADO` (RF-006) → nasce **no contrato** (RF-012, V6.2);
  no V6.1 não se cria status financeiro no cliente (evita redundância). Ver §5.2.

## 4. Arquitetura

Segue os padrões já estabelecidos no projeto:

- **Banco:** nova migration `supabase/migrations/0026_financeiro_cadastros.sql`, idempotente
  (`create table if not exists`, `drop policy if exists ... ; create policy ...`), aplicada pelo
  runner próprio (`npm run db:migrate`). Migrations anteriores são imutáveis.
- **Permissões:** fonte única em `src/lib/financeiro/permissoes.ts`, espelhando a RLS (mesmo padrão
  de `src/lib/clientes/permissoes.ts`).
- **UI + Server Actions:** área nova `src/app/(app)/financeiro/cadastros/` (App Router, Next 16),
  uma sub-rota por cadastro, com Server Actions co-localizadas.
- **Testes:** RLS via `scripts/db-test-rls.mjs` (`npm run db:test`); unitários em
  `src/tests/financeiro/` (Vitest).

Convenções de tabela (RNF-005): PK `uuid default gen_random_uuid()`; monetário `numeric(15,2)`;
auditoria mínima `criado_em timestamptz default now()`, `criado_por uuid references usuarios(id)`,
`atualizado_em`, `atualizado_por`; inativação lógica `ativa boolean not null default true`.

## 5. Modelo de dados

### 5.1 Novos enums

```
financeiro_conta_tipo   : CORRENTE | POUPANCA | CAIXA | CARTAO
categoria_natureza      : RECEITA | DESPESA
categoria_grupo         : OPERACIONAL | NAO_OPERACIONAL
faixa_faturamento       : ATE_81K | ATE_360K | ATE_4_8MI | ATE_78MI | ACIMA_78MI
```

Faixas de faturamento alinhadas aos limites fiscais usuais: `ATE_81K` (teto MEI), `ATE_360K` e
`ATE_4_8MI` (faixas do Simples Nacional), `ATE_78MI` (teto do Lucro Presumido), `ACIMA_78MI`.

### 5.3 Seeds na migration

- **Plano de contas padrão (RF-004):** Receitas — Honorários mensais, Honorários eventuais, 13º
  honorário, Outras receitas. Despesas — Pessoal e encargos, Softwares e licenças, Ocupação,
  Impostos, Marketing, Serviços de terceiros, Financeiras. (1º nível; editável.)
- **Centros de custo (RF-005):** Fiscal, Contábil, Pessoal/DP, Legalização, Consultoria,
  Administrativo.

Seeds idempotentes (`insert ... where not exists` / `on conflict do nothing` com chave estável).

### 5.4 Extensão financeira do cliente (RF-006, RF-007)

`alter table clientes_financeiro add column if not exists`:
| Campo | Tipo | Regra |
|---|---|---|
| dia_vencimento | smallint | check entre 1 e 28 (RF-007) |
| qtd_funcionarios | int | ≥ 0 |
| faixa_faturamento | faixa_faturamento | |
| data_saida | date | Alimenta churn básico |

**Reaproveitados (NÃO duplicar):** `clientes.regime_tributario` (enum já existente e mais rico:
Simples/Presumido/Real/MEI/Isento-PF, com constraint tipo×regime), `clientes.data_inicio` (≈ data de
entrada), `clientes.status` (comercial `ativo|inativo`).

**Status financeiro (RF-006 ATIVO/SUSPENSO/ENCERRADO):** não é criado no V6.1. O ciclo
suspenso/encerrado nasce no **contrato** (RF-012, V6.2); criá-lo também no cliente seria redundante.
`data_saida` cobre o churn básico até lá.

## 6. Permissões e RLS

### 6.1 Módulo `src/lib/financeiro/permissoes.ts`

```ts
// Cria/edita/inativa todos os cadastros de apoio.
podeGerenciarFinanceiro(papel): admin | financeiro
// Lê catálogos que o contador precisa p/ lançar honorários eventuais (V6.3).
podeLerCatalogo(papel): admin | financeiro | contador
```

### 6.2 RLS por tabela (via `auth_papel()`)

| Tabela | SELECT | INSERT / UPDATE |
|---|---|---|
| conta_bancaria | admin, financeiro | admin, financeiro |
| centro_custo | admin, financeiro | admin, financeiro |
| fornecedor | admin, financeiro | admin, financeiro |
| **categoria** | admin, financeiro, **contador** | admin, financeiro |
| **servico** | admin, financeiro, **contador** | admin, financeiro |
| clientes_financeiro | *(inalterada)* admin+financeiro; contador só os seus | *(inalterada)* |

- **Assistente:** sem policy em nenhuma tabela financeira → não vê nada (igual ao honorário hoje).
- A extensão de `clientes_financeiro` **não altera** as policies existentes — só adiciona colunas.
- Sem exclusão física: não há policy de DELETE; a inativação é via `ativa = false`.

## 7. UI e navegação

Área nova **`/financeiro/cadastros`** com sub-páginas:
- **Contas bancárias** — lista + form (nome, tipo, banco/agência/número, saldo inicial + data) + inativar.
- **Plano de contas** — árvore de 2 níveis; form (nome, natureza, grupo, pai, ordem DRE) + inativar.
- **Centros de custo** — lista + form (nome) + inativar.
- **Fornecedores** — lista + form (nome, CNPJ/CPF validado, contato, categoria padrão) + inativar.
- **Serviços** — lista + form (nome, descrição, preço sugerido, categoria de receita) + inativar.

Server Actions co-localizadas por cadastro em
`src/app/(app)/financeiro/cadastros/<cadastro>/actions.ts`, com gate `podeGerenciarFinanceiro` no
servidor (trava efetiva além da RLS). Entrada de menu visível só a quem tem permissão.

A **extensão do cliente** (dia de vencimento, qtd. funcionários, faixa de faturamento, data de saída)
aparece como campos na **ficha do cliente já existente**, dentro da área financeira dela (mesmo gate
`podeVerHonorario` que já protege o honorário).

## 8. Testes

- **RLS** (`scripts/db-test-rls.mjs`): matriz admin/financeiro/contador/assistente para cada tabela
  nova. Casos-chave: contador **lê** `categoria`/`servico` mas **não** `conta_bancaria`; contador
  **não escreve** em nenhuma; assistente não vê nada; financeiro gerencia tudo.
- **Unitários** (`src/tests/financeiro/permissoes.test.ts`): tabela-verdade de
  `podeGerenciarFinanceiro` e `podeLerCatalogo` para os 4 papéis.
- Validação de `cnpj_cpf` do fornecedor reusa o validador existente do módulo de clientes.
- Antes de commitar: `npm run lint && npm run typecheck && npm test && npm run db:migrate && npm run db:test`.

## 10. Riscos e observações

- **Redundância cliente × contrato:** manter status suspenso/encerrado só no contrato (V6.2) exige
  disciplina para não recriá-lo no cliente depois. Documentado aqui como decisão consciente.
- **Hierarquia do plano de contas:** limitada a 2 níveis por RF-003; o app impede aninhar além disso.
- **Seeds editáveis:** o plano de contas e os centros de custo padrão são pontos de partida; o usuário
  pode renomear/inativar. Seeds idempotentes não sobrescrevem edições do usuário em re-execução.
