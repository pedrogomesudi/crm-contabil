# V8.2a — Rollout SALDO: Linguagem de Design + Auth + Clientes — Design

**Data:** 2026-07-04
**Marco:** V8 — Layout e estética · **Sub-marco:** V8.2a (1ª fatia do rollout)
**Depende de:** V8.1 (tokens, fontes, `LogoSaldo`, primitivos base). Referência visual aprovada: mockup de Clientes (estilo SALDO/Apple).

## 1. Objetivo

Iniciar o rollout do design system com **recomponentização profunda** (estilo Apple — limpo, técnico, funcional): (a) fixar a **linguagem de design** em princípios reutilizáveis, (b) **ampliar os primitivos** de UI para cobrir formulários e listas, e (c) aplicar tudo às áreas de **Autenticação** e **Clientes** como implementação de referência para as próximas fatias (V8.2b financeiro, V8.2c NFS-e, V8.2d resto).

## 2. Fronteiras (escopo)

**Dentro:** princípios de design (documentados em `docs/`), primitivos novos (`Campo`, `Input`/`Select`/`Textarea`, `Painel`, `Chip`, `Toolbar`, `EmptyState`, `Iniciais`, `Badge` de regime/status), e recomponentização das telas de **auth** e **clientes** (lista, novo, ficha, formulários e componentes de clientes).

**Fora (próximas fatias):** financeiro, NFS-e, atendimento, integrações/Domínio, documentos/assinatura, usuários — só migram nas V8.2b/c/d. Sem mudança de banco/funcionalidade (é visual). Sem novas dependências.

## 3. Linguagem de design SALDO (princípios)

Documentar em `docs/design/saldo-ui.md` (guia curto reusado pelas fatias seguintes):
1. **Restrição.** Fundo creme, cards brancos, **hairlines** (`border-linha` 1px), cantos arredondados (`rounded-2xl` cards, `rounded-lg`/`xl` controles). Sombra sutil só onde eleva (painéis).
2. **Hierarquia tipográfica.** Títulos/números em **`font-display`** (Space Grotesk) com tracking negativo; corpo/UI em `font-sans`; **dados** (CNPJ, R$, códigos, rótulos-eyebrow) em **`font-mono`**. Números que alinham → `tabular-nums`.
3. **Espaço e ritmo.** Respiro generoso; agrupar por `gap` (flex/grid), não margens soltas. Densidade média (linhas de tabela ~13px padding vertical).
4. **Cor com significado.** Verde = primária/positivo; violeta = dados/destaque; negativo = erro/inativo; neutros para o resto. O acento não é decorativo — marca ação e estado.
5. **Estado legível num relance.** Situação em **pill** (bolinha + texto), regime em **badge** colorido, valores em mono alinhados à direita.
6. **Interatividade evidente.** Hover sutil em linhas/botões; **foco-visível** verde (WCAG) preservado; alvos ≥ 36px.
7. **Vazio ≠ erro.** `EmptyState` (ícone + frase + ação), distinto de mensagens de erro (`role="alert"`, cor negativo).
8. **Responsivo.** Colunas secundárias somem no mobile; toolbars quebram; sem scroll horizontal do corpo (tabelas em `overflow-x-auto`).

## 4. Primitivos novos (`src/components/ui/`)

- **`Campo`** — `rotulo` + `children` (o controle) + `hint?`/`erro?`. Renderiza `<label>` (mono/eyebrow discreto) e a mensagem de erro (negativo). Envolve os inputs.
- **`Input` / `Select` / `Textarea`** — estilos padronizados (borda linha, bg surface, foco verde, placeholder cinza). Repassam `...props` nativos (compatíveis com Server Actions e formulários controlados via `value/onChange`).
- **`Painel`** — contêiner de lista/tabela: `rounded-2xl border border-linha bg-white overflow-hidden shadow-sutil`; slots opcionais de header/footer.
- **`Chip`** — pílula clicável (filtro): estados `on` (fundo tinta, texto creme) / off (borda linha). Para toolbars.
- **`Toolbar`** — faixa com busca (input com ícone) + `children` (chips/ações), responsiva.
- **`EmptyState`** — `icone?`, `titulo`, `descricao?`, `acao?`.
- **`Iniciais`** — quadradinho arredondado com as 2 primeiras iniciais (fundo `verde/10`, texto verde); helper puro **`iniciais(nome)`** (testável).
- **`Badge`** (estender) — variantes de **regime** (`simples`=verde, `presumido`=violeta, `real`=cinza, `mei`=âmbar) e **situação** via pill. Helper puro **`badgeRegime(regime)`** → variante (testável).

- **Formulários** (`FormCliente`, `HonorarioForm`, `novo`): inputs → `Campo`/`Input`/`Select`; botões → `Botao`; "Buscar na Receita" e demais ações mantêm comportamento, só re-skin.

## 7. Testes e verificação
- **Unit:** `iniciais(nome)` (2 iniciais; nome único; vazio → "?") e `badgeRegime(regime)` (mapa regime→variante) — puros. Smoke `renderToStaticMarkup` de `Campo`/`EmptyState`/`Iniciais` (renderiza sem erro; contém o rótulo/iniciais).
- **Verificação:** `npm run lint && npm run typecheck && npm test && npm run build` (verde) + **checagem visual** desktop/mobile das telas de auth e clientes (foco-visível, acentos, tabela, formulários, estados vazio/erro).
- Sem migration; RLS/permissões/queries **inalteradas** (só apresentação).

## 8. Riscos e observações
- **Regressão funcional:** o maior risco é quebrar formulários controlados (`FormCliente` é controlado com estado). Regra: **trocar só classes/wrappers visuais**, preservar `name`/`value`/`onChange`/actions.
- **Consistência transitória:** financeiro/NFS-e/etc. seguem no estilo antigo até as próximas fatias — esperado.
- **Reuso:** os primitivos e o guia `saldo-ui.md` são a base das V8.2b/c/d — investir na qualidade aqui acelera o resto.
- **Acessibilidade:** manter `aria-*`, `role="alert"` nos erros, labels associadas (o `Campo` liga `<label>` ao controle via `htmlFor`/`id`).
