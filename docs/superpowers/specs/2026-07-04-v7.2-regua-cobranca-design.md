# V7.2 — Régua de Cobrança Automática — Design

**Data:** 2026-07-04
**Marco:** V7 — Integração WhatsApp · **Sub-marco:** V7.2 (régua automática)
**Depende de:** V7.1 (cliente Z-API `zapi.ts`, `mensagem.ts` templates/telefone, `whatsapp_mensagem`, config cifrada), V6.2 (titulo/baixa/saldo), clientes_financeiro.
**Requisitos-fonte:** `módulo financeiro.docx` RF-029, RF-032, RF-033; roadmap V7.2.

## 1. Objetivo

Automatizar a cobrança dos honorários por WhatsApp: **etapas configuráveis** (D-3, D+1, D+7, D+15) que disparam sozinhas sobre os títulos a receber, com **opt-out por cliente** (LGPD), **idempotência** (uma etapa por título) e **histórico**. Reusa a fundação de envio da V7.1.

## 2. Fronteiras (escopo)

**Dentro:** etapas configuráveis (CRUD), motor de seleção/envio, opt-out por cliente, idempotência por `(titulo, etapa)`, rota protegida de disparo + botão manual, toggle liga/desliga, tela de gestão, histórico.

**Fora (futuro):** canal e-mail (só WhatsApp aqui), atendimento/inbox (V7.3), negociação/parcelamento pelo bot, mídia/boleto anexo, relatórios de efetividade da cobrança.

**Ressalvas:** herda o risco do Z-API não-oficial (número dedicado). Sem `pg_net`/`http` no banco → o disparo roda **na app** (rota), acionada por agendador externo.

## 3. Modelo de dados

### 3.1 `regua_etapa` (configurável)
| Campo | Tipo | Regra |
|---|---|---|
| id | uuid PK | |
| nome | text not null | ex.: "Lembrete D-3" |
| dias_offset | int not null | dias relativos ao vencimento; **negativo = antes**, **positivo = depois** (0 = no dia) |
| template | text not null | com `{nome}`, `{valor}`, `{vencimento}`, `{dias}` |
| ordem | int not null default 0 | exibição |
| ativa | boolean not null default true | |
| auditoria | (padrão) | |
Único: `unique (dias_offset) where ativa` — evita duas etapas ativas no mesmo dia (ambiguidade). **Seed:** D-3 (lembrete), D+1, D+7, D+15.

### 3.2 Opt-out por cliente
`alter table clientes_financeiro add column cobranca_whatsapp boolean not null default true`. `true` = entra na régua; `false` = excluído (opt-out).

### 3.3 Idempotência (reusa `whatsapp_mensagem`)
`alter table whatsapp_mensagem add column etapa_id uuid references regua_etapa(id) on delete set null`.
Índice único parcial: `unique (titulo_id, etapa_id) where etapa_id is not null` — cada etapa dispara **uma vez** por título (mesmo em reprocessamento).

### 3.4 Toggle da régua
`alter table whatsapp_config add column regua_ativa boolean not null default false`. A rota automática só age com `regua_ativa = true`; o botão manual ignora o flag (ação humana explícita).

## 4. Motor (`src/lib/whatsapp/regua.ts` + action)

### 4.1 Matcher puro (testável) — `etapaDoDia`
`etapaDoDia(etapas: EtapaAtiva[], hoje: string, vencimento: string): EtapaAtiva | null`
- `dias = diffDias(hoje, vencimento)` = (hoje − vencimento) em dias (positivo = vencido, negativo = a vencer).
- retorna a etapa ativa cujo `dias_offset === dias`, ou null.
- `diffDias(a, b)` puro (datas ISO, UTC, sem horário).

### 4.2 `processarRegua(hoje, { forcarManual })` (server-side, service_role)
Para cada título **RECEBER** com `status in ('ABERTO','BAIXADO_PARCIAL')` e **saldo > 0**:
1. cliente **não** optou por sair (`clientes_financeiro.cobranca_whatsapp = true`) e **tem telefone** válido;
2. `etapa = etapaDoDia(etapasAtivas, hoje, titulo.vencimento)`; se null → pula;
3. já enviada? (existe `whatsapp_mensagem` com `titulo_id + etapa_id`) → pula;
4. monta `aplicarTemplate(etapa.template, {nome, valor, vencimento, dias})`, `enviarTexto` (Z-API), grava `whatsapp_mensagem` com `etapa_id` e status.
Retorna resumo `{ processados, enviados, pulados, erros }`. A inserção do log com o índice único é a **trava final de idempotência** (mesmo em corrida).

## 5. Disparo e agendamento

- **Rota** `POST /api/cron/regua-cobranca` (route handler): valida `Authorization: Bearer $CRON_SECRET` (env). Se `whatsapp_config.regua_ativa` → chama `processarRegua(hoje)`. Usa **service_role** (bypassa RLS; roda fora de sessão de usuário). Sem config Z-API ou `CRON_SECRET` → responde erro claro sem quebrar.
- **Agendador externo** (setup único do usuário): task agendada no EasyPanel **ou** cron-job.org, diário ~08:00, `POST` na rota com o header do segredo. Documentado no fechamento.
- **Botão manual** "Processar régua agora" (tela, admin/financeiro) → action `dispararReguaManual()` → `processarRegua(hoje, {forcarManual:true})` (ignora `regua_ativa`).

## 6. UI
- **`/financeiro/regua-cobranca`** (gate `podeGerenciarFinanceiro`): toggle **régua ativa**; **CRUD de etapas** (nome, dias_offset, template, ativa) reusando o padrão `CadastroCrud`/form simples; botão **Processar agora** (mostra o resumo); **histórico recente** (`whatsapp_mensagem` com etapa/cliente/status).
- **Ficha do cliente:** toggle **"Receber cobrança por WhatsApp"** (opt-out) — grava `clientes_financeiro.cobranca_whatsapp`. Gate `podeGerenciarFinanceiro`.
- Link "Régua de cobrança" no hub `/financeiro/cadastros`.

## 7. Permissões, RLS e segurança
- `regua_etapa`: RLS admin/financeiro (SELECT/INSERT/UPDATE).
- `whatsapp_config.regua_ativa`: admin (já é config admin).
- opt-out: escrito por `podeGerenciarFinanceiro` (é dado financeiro do cliente).
- **Rota de cron:** protegida por `CRON_SECRET` (env, nunca `NEXT_PUBLIC_`); usa service_role; compara o segredo em tempo constante.
- Reusa o cliente Z-API (credenciais cifradas, V7.1).

## 8. Testes
- **Unit (`src/tests/whatsapp/regua.test.ts`):** `diffDias` (a vencer negativo, vencido positivo, mesmo dia 0); `etapaDoDia` (casa offset exato; sem etapa → null; ignora inativa); `aplicarTemplate` com `{dias}`.
- **RLS (`db:test`):** contador/assistente não gerenciam `regua_etapa`; unique parcial `(titulo_id, etapa_id)` barra duplicata; unique `(dias_offset) where ativa` barra etapa ativa repetida.
- **Manual:** ligar a régua, criar um título vencendo hoje±offset, "Processar agora", conferir envio + log + idempotência (rodar 2×, envia 1×).
- Antes de commitar: `npm run lint && npm run typecheck && npm test && npm run db:migrate && npm run db:test`.

## 10. Riscos e observações
- **Match exato por dia:** a etapa dispara quando `dias == dias_offset`. Se o cron **falhar num dia**, aquela etapa é pulada para os títulos daquele dia (as demais etapas seguem). Aceitável; o botão manual permite recuperar rodando no dia. (Catch-up de etapas perdidas fica como melhoria futura.)
- **Ordem do envio × baixa:** o motor lê saldo no momento; um pagamento registrado antes do disparo tira o título da régua (saldo 0). Correto.
- **Idempotência:** garantida pelo índice único `(titulo_id, etapa_id)` — a corrida perde a segunda inserção, não duplica o envio lógico (o envio ocorre só se a checagem prévia passou; em corrida rara pode haver 2 envios físicos antes do commit — mitigado por rodar single-scheduler).
- **Volume:** envios sequenciais com o timeout do Z-API; para carteiras grandes, processar em lote é melhoria futura (hoje ~dezenas de clientes).
- **LGPD:** opt-out respeitado no motor; texto e opt-out documentados. Base legal = execução de contrato/legítimo interesse (relação de honorários existente).
