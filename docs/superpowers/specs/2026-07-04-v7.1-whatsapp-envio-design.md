# V7.1 — Envio WhatsApp (fundação) — Design

**Data:** 2026-07-04
**Marco:** V7 — Integração WhatsApp · **Sub-marco:** V7.1 (fundação)
**Depende de:** clientes (telefone), títulos (contas a receber, V6.2), `cripto.ts` (V5).
**Requisitos-fonte:** `módulo financeiro.docx` RF-030 (parcial — envio WhatsApp), roadmap V7.

## 1. Objetivo

Entregar a base de envio de WhatsApp: integração com o **Z-API** (provedor não-oficial), tela de configuração das credenciais, cliente de envio isolado, templates de mensagem e **envio pontual de cobrança** a partir de um título de contas a receber, com histórico. Essa fundação é reusada pela régua automática (V7.2).

## 2. Decomposição da V7

| Sub-marco | Escopo |
|---|---|
| **V7.1 — Envio (esta fatia)** | Config + cliente Z-API + templates + envio pontual de cobrança + histórico |
| V7.2 — Régua de cobrança | Etapas D-3/D+1/D+7/D+15 automáticas sobre títulos vencidos (reusa V7.1) |
| V7.3 — Atendimento | Inbox bidirecional (webhook de entrada + caixa de entrada) |

## 3. Fronteiras (escopo V7.1)

**Dentro:** config Z-API (credenciais cifradas), cliente `enviarTexto`/`statusConexao`, normalização de telefone, substituição de templates, envio pontual de cobrança de um título + histórico `whatsapp_mensagem`.

**Fora (V7.2+):** régua automática/agendamento, atendimento/inbox, templates com CRUD editável (V7.1 usa textos padrão + variáveis), mídia/anexos, botões interativos.

**Ressalva:** o Z-API é **não-oficial** (usa WhatsApp Web) — risco de banimento do número existe; usar um número dedicado do escritório. Documentar na tela de config.

## 4. Arquitetura

- **Provedor abstraído:** `src/lib/whatsapp/zapi.ts` concentra as chamadas Z-API. Trocar de provedor depois = mudar só esse arquivo. Cliente puro (recebe config, faz `fetch`).
- **Config cifrada:** credenciais sensíveis (`token`, `client_token`) cifradas com `cripto.ts` (AES-256-GCM) usando a env `WHATSAPP_CRIPTO_KEY` (definir no EasyPanel, como a `NFSE_CERT_KEY`). O `instance` (não-secreto) em claro.
- **Envio server-side:** as chamadas HTTP ao Z-API rodam em Server Action (nunca no cliente). `User-Agent` explícito não é necessário (Z-API não bloqueia como a BrasilAPI).

## 5. Modelo de dados

| Campo | Tipo | Regra |
|---|---|---|

| instance | text | ID da instância Z-API (não-secreto) |
| token_cifrado | text | token da instância, cifrado |
| client_token_cifrado | text | token de segurança da conta, cifrado |
| atualizado_em/por | | |
RLS: **admin** (`auth_papel() = 'admin'`) — mesma regra da config NFS-e.

### 5.2 `whatsapp_mensagem` (histórico)
| Campo | Tipo | Regra |
|---|---|---|
| id | uuid PK | |
| cliente_id | uuid → clientes | opcional |
| titulo_id | uuid → titulo | opcional (cobrança de um título) |
| telefone | text | normalizado (só dígitos) |
| texto | text | conteúdo enviado |
| status | enum `ENVIADO`\|`ERRO` | resultado |

RLS: **admin/financeiro** tudo; **contador** só as dos seus clientes (via `cliente_id`); assistente nada.

## 6. Cliente Z-API (`src/lib/whatsapp/zapi.ts`)
- `type ZapiConfig = { instance: string; token: string; clientToken: string }`.
- `enviarTexto(cfg, telefone, texto): Promise<{ ok: boolean; erro?: string; resposta?: unknown }>`
  - `POST https://api.z-api.io/instances/{instance}/token/{token}/send-text`
  - headers: `{ "Content-Type": "application/json", "Client-Token": clientToken }`
  - body: `{ phone: telefone, message: texto }`
  - timeout (AbortController); trata não-2xx / rede → `{ ok:false, erro }`.
- `statusConexao(cfg): Promise<{ conectado: boolean; erro?: string }>`
  - `GET .../status` com `Client-Token`; interpreta `connected`.

## 7. Helpers puros (`src/lib/whatsapp/mensagem.ts`)
- `normalizarTelefone(bruto: string): string | null` — só dígitos; garante DDI 55 (adiciona se faltar, para números BR com 10–11 dígitos); retorna null se inválido (< 12 ou > 13 dígitos após o 55).
- `aplicarTemplate(template: string, vars: Record<string,string>): string` — substitui `{chave}` por valor (chaves ausentes viram "").
- `TEMPLATES` — textos padrão: `cobranca` ("Olá {nome}, consta em aberto o valor de {valor} com vencimento em {vencimento}…") e `aviso_vencimento`.

## 8. Server Actions (`src/app/(app)/configuracoes/whatsapp/actions.ts` + no título)
- **Config:** `salvarConfigWhatsapp(fd)` (gate admin) — cifra `token`/`client_token`, upsert id=1. `testarConexao()` (gate admin) — decifra, chama `statusConexao`.
- **Envio de cobrança:** `cobrarViaWhatsapp(tituloId)` (gate `podeVerHonorario`) — carrega o título + cliente (telefone, razão), monta a mensagem via `aplicarTemplate(TEMPLATES.cobranca, {nome, valor, vencimento})`, normaliza o telefone, decifra a config, `enviarTexto`, e **grava `whatsapp_mensagem`** com o status. Erros (sem telefone, sem config, falha Z-API) retornam mensagem clara e gravam status ERRO.

## 9. UI
- **`/configuracoes/whatsapp`** (admin): form com instance/token/client-token (os cifrados mostram só "configurado"), botão **Testar conexão** (mostra conectado/erro), aviso sobre número dedicado + risco do não-oficial.
- **Contas a receber** (`ContasReceber`): botão **"Cobrar (WhatsApp)"** por título em aberto → `cobrarViaWhatsapp` → feedback (enviado/erro). Só aparece se o cliente tem telefone.
- Link "WhatsApp" no menu/hub de configurações.

## 10. Permissões e RLS
- Config: admin (`podeConfigurarWhatsapp(papel) = papel==='admin'`, novo helper).
- Envio/histórico: `podeVerHonorario` (admin/financeiro/contador-dono). RLS de `whatsapp_mensagem` escopa o contador.

## 11. Testes
- **Unit (`src/tests/whatsapp/`):** `normalizarTelefone` (11 díg → 55+11; já com 55; inválidos → null), `aplicarTemplate` (substitui/ausente vazio), montagem do payload/URL do Z-API (sem fazer fetch real — testar a função que monta URL/body se extraída).
- **Manual (com credenciais):** conectar a instância, testar conexão, cobrar um título real e conferir a entrega + o log.

## 13. Riscos e observações
- **Z-API não-oficial:** risco de ban do número (WhatsApp Web). Usar número dedicado; documentar. Trocar de provedor é isolado em `zapi.ts`.
- **Credenciais:** cifradas com `WHATSAPP_CRIPTO_KEY` (nova env no EasyPanel). Sem a env, config falha com erro claro.
- **Sem número/telefone do cliente:** o envio informa e não quebra; grava ERRO no log.
- **Entrega real** só testável com credenciais + instância conectada (fora do CI).
