# RF-072 — Rentabilidade por segmento — Fatia B (porte) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Criar o campo **porte** (MEI/ME/EPP/Demais) no cadastro do cliente e ativar o agrupamento da rentabilidade por porte (o seletor já lista "Porte" desde a Fatia A).

**Architecture:** Enum + coluna em `clientes`; `PORTES` em `tipos.ts`; `porte` no `clienteSchema` (Zod) para fluir do form ao banco; `<select>` no `FormCliente`; `porte` em `LinhaRentab` + no select da ação de rentabilidade. A agregação e a tela já existem (Fatia A).

**Tech Stack:** Next.js 16, TypeScript, Supabase (Postgres enum), Zod, Tailwind 4, vitest.

## Global Constraints

- Migration imutável/idempotente (enum via `do $$ ... exception when duplicate_object`; `add column if not exists`).
- `PORTES` deve bater EXATAMENTE com os valores do enum `porte_empresa` do banco.
- Guard `divida-ui`: sem `border` estático em input escrito à mão (usar `controleCls`); sem `←`/`amber-\d`.
- `package.json.version` sobe com o CHANGELOG no mesmo PR; `versao.test.ts` exige que batam.
- Rodar antes de commitar: `npm run lint && npm run typecheck && npm test && npm run format && npm run build`.
